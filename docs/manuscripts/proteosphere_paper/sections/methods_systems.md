# Methods and Systems

ProteoSphere is designed as a warehouse-first evidence system for biomolecular interaction machine learning. The central idea is to keep a compact, release-pinned reference library in front of the heavy raw estate so dataset design, provenance review, and leakage checks can be done without repeatedly reopening the full upstream corpus. In the current architecture, the library holds the selection surface, lineage-preserving canonical records, and compact planning data, while coordinates, maps, alignments, long assay tables, and other bulky payloads remain deferred until a candidate is actually selected (see [source_storage_strategy.md] and [lightweight_reference_library_master_plan.md]).

The source contract is release-aware rather than live-web aware. The release matrix defines the primary identity and evidence spine as UniProt for sequence identity, RCSB/PDBe and AlphaFold DB for structure, BindingDB/BioGRID/IntAct for interaction and assay evidence, InterPro and related motif systems plus DisProt for annotation, and Reactome and EMDB for pathway and structural context; the evolutionary corpus remains the explicitly pending acquisition lane (see [source_release_matrix.md]). The warehouse summary reports `best_evidence` as the default view and confirms that the current warehouse metadata sync is ready (see [warehouse_summary.json]). The manifest currently materializes 13 family tables, including proteins, protein_variants, pdb_entries, structure_units, ligands, protein_ligand_edges, protein_protein_edges, motif_domain_site_annotations, pathway_roles, provenance_claims, materialization_routes, leakage_groups, and similarity_signatures (see [warehouse_manifest.json]).

The compactness of the library comes from storage discipline rather than from dropping provenance. The storage policy separates six tiers: a pinned manifest, a planning index, a canonical store, a feature cache, deferred fetch/materialization, and a narrow scrape-only enrichment lane (see [source_storage_strategy.md]). That design means the library can keep accession-first routing data and lineage-preserving summaries hot, while heavy payloads such as raw mmCIF files, map volumes, full PSI-MI/XML evidence exports, or full alignment bundles stay lazy until needed. Supplemental scraping is deliberately constrained to accession-scoped portal or HTML enrichments, including InterPro, PROSITE, ELM, DisProt, Reactome, and RCSB motif-search payloads, and those captures remain enrichment material rather than primary identity (see [source_storage_strategy.md]).

The runtime code follows the same philosophy. `reference_library.py` exposes a default `best_evidence` claim view through `extract_claim_view()`, while still allowing `raw` and `derived_or_scraped` projections when a workflow explicitly asks for them. The same module also exposes `build_entity_conflict_summary()`, which reports whether the underlying record contains conflicts across claim surfaces and falls back to a summary-contract interpretation when the row is a compact warehouse summary rather than a fully materialized claim object (see [reference_library.py]). In practice, this means the library is not forcing all available evidence into one row; it is preserving alternate claims in a controlled way and surfacing the most defensible view by default.

Route resolution is also registry-driven rather than ad hoc. `resolve_materialization_route()` normalizes the source-family name, anchors the route against the source registry, prefers promoted registry entries when multiple matches exist, and distinguishes library-owned pointers from external pointers before deciding whether a route is already resolved or still needs anchoring (see [reference_library.py] and [source_registry.json]). That keeps the materialization contract tied to release-stamped source families instead of to mutable local paths.

At the training-set and benchmark layer, the Studio/runtime code uses the warehouse as a first gate. `_custom_dataset_warehouse_catalog_path()` checks `PROTEOSPHERE_WAREHOUSE_ROOT` first and then the local default warehouse roots, so the runtime can be pointed at the current reference library without rewriting the downstream code path (see [runtime.py]). When candidate rows are converted into benchmark rows, `_custom_manifest_record_to_row()` carries accession grounding, custom-manifest identifiers, and UniRef cluster hints into the row metadata so the candidate can be audited against the warehouse rather than treated as an isolated CSV row (see [runtime.py]). The split compiler then calls `_build_leakage_summary()`, which computes the exact overlap in protein accessions between train-plus-validation and test rows and blocks the split when direct protein overlap is present; the generated report surfaces the same gate in plain language (see [runtime.py]). This is intentionally a conservative first gate rather than a full replacement for deeper family- or structure-level auditing.

The literature-review side of ProteoSphere was calibrated against a fully source-bundled Tier 1 audit corpus rather than against abstract examples. The current proof set contains 29 confirmed Tier 1 hard-failure papers, 18 of which were published in 2023 or later, and the local source bundle includes 29/29 Tier 1 article PDFs plus 119 supplemental or evidence items for reconstruction and audit use (see [literature_hunt_tier1_master_summary.md], [literature_hunt_recent_expansion.md], and [literature_hunt_tier1_source_bundle.md]). That corpus was used to verify that the reviewer can separate paper-specific failures, inherited benchmark-family failures, and mitigation-aware controls without treating every audited paper as equally broken.

The strongest current limitation is scope fidelity, not operational utility. The condensed warehouse is operationally effective and reproduces top-level paper outcomes on the current validation set, but it is not yet a full evidence-equivalent replacement for the broader downloaded collection (see [library_vs_broad_collection_equivalence.md]). The validation report states that the library-backed evaluator matches 20 of 20 top-level paper statuses, while also noting that the broader collection still preserves richer roster reconstruction and structure-level overlap evidence for some cases. For that reason, ProteoSphere treats the compact warehouse as the primary truth surface for normal evaluation and retains the broader source bundle as a governed fallback for provenance recovery, paper-faithful audits, and deep-dive structure checks.

Taken together, these choices make ProteoSphere a compact evidence warehouse rather than a miniature raw corpus. The system is built to answer four questions quickly and truthfully: what the object is, what is biologically close to it, what evidence supports it, and whether it is safe to include in a train/test split without leaking identity, family, partner, or structure context.
